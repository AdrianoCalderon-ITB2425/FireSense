# Comparison: Databases and Data Management

---

## MongoDB vs CouchDB

| Criterion | **MongoDB** | CouchDB |
|---|---|---|
| License | SSPL (Server Side Public License) | Apache 2.0 |
| Current version | 7.0.x | 3.x |
| Data model | BSON Documents (binary JSON) | JSON Documents |
| Queries | MQL (MongoDB Query Language), very powerful | Mango Query or MapReduce, more limited |
| Replication | Native Replica Sets | Native multi-master replication |
| Horizontal scalability | Native sharding | Limited compared to MongoDB |
| Indexes | Very comprehensive (compound, geospatial, text) | Basic |
| RAM consumption | ~300–600 MB | ~100–200 MB |
| Ecosystem / drivers | Very broad (all languages) | More limited |
| Community | Very large | Small |
| Main use case | General applications with complex data | Offline synchronization, distributed replication |

### Justification for the choice: MongoDB

- **Ecosystem and documentation**: MongoDB is the most widely used document database in the world, with official drivers for all languages and a massively active community. CouchDB has a considerably smaller community.
- **Query capabilities**: MongoDB's MQL is significantly more powerful and flexible than CouchDB's query system, making it easier to filter alerts, sensor configurations, and system metadata.
- **Integration with the existing stack**: MongoDB has native and well-documented integration with Node-RED, reducing configuration complexity.
- **Future scalability**: Although the data volume for this project is small, MongoDB offers native sharding if the project scales, whereas CouchDB has limitations in this regard.
- **Important note**: As mentioned in the project planning, MongoDB is optional in FireSense. If all data consists of sensor time series, InfluxDB may be sufficient. MongoDB would only add value for storing historical alerts, sensor configurations, and non-temporal metadata.

## InfluxDB vs TimescaleDB

| Criterion | **InfluxDB** | TimescaleDB |
|---|---|---|
| License | MIT (v1.8) / BSL 1.1 (v2.x+) | Apache 2.0 (Community) / Timescale License |
| Current version | 2.7.x / 1.8.x (LTS) | 2.x |
| Technological base | Own engine (TSM) | PostgreSQL extension |
| Query language | Flux (v2) / InfluxQL (v1) | Standard SQL |
| Time-series optimization | Native, engine designed for this | Very good, but inherited from PostgreSQL |
| Data compression | High (optimized TSM engine) | High (chunk compression) |
| RAM consumption | ~200–500 MB | ~300–600 MB + PostgreSQL overhead |
| Grafana integration | Official plugin, first-class | Official plugin available |
| Node-RED integration | Direct native plugin | Via PostgreSQL connector |
| Retention policies | Native and simple | Available (data retention policies) |
| Learning curve | Medium (Flux is new) | Low if SQL is known |

### Justification for the choice: InfluxDB

- **Native design for time series**: InfluxDB was built from the ground up to store and query timestamped data (metrics, sensor readings). TimescaleDB is an excellent extension, but it remains PostgreSQL with added functions, which implies higher resource overhead.
- **Direct integration with the MING stack**: The Mosquitto → Node-RED → InfluxDB → Grafana combination is a widely documented and proven architecture. Specific documentation exists for this stack in industrial and academic IoT projects.
- **Simple retention policies**: InfluxDB has a native data retention policy system (how long to keep readings) that is trivial to configure, which is very useful for managing disk space in a lab environment.
- **Version 1.8 as a lightweight option**: For environments with limited resources, InfluxDB v1.8 (still maintained as LTS) consumes significantly fewer resources than v2.x, offering a fallback option if IsardVDI VMs have limitations.
- **Native Grafana datasource**: The official InfluxDB plugin in Grafana is one of the most mature and comprehensive on the platform.

## PostgreSQL vs MySQL / MariaDB

| Criterion | **PostgreSQL** | MySQL / MariaDB |
|---|---|---|
| License | PostgreSQL License (free, similar to BSD) | GPL 2.0 (MySQL) / GPL 2.0 (MariaDB) |
| Current version | 16.x | MySQL 8.x / MariaDB 11.x |
| Type | Object-relational database | Relational database |
| ACID compliance | Full | Full (InnoDB) |
| Native JSON support | Very advanced (jsonb, indexes on JSON) | Basic (MySQL 8+) / Basic (MariaDB) |
| Advanced data types | UUID, arrays, hstore, geographic types, enums | Fewer native types |
| Stored procedures | PL/pgSQL, PL/Python, PL/Perl... | SQL + limited |
| Scalability | Very high (partitioning, logical replication) | High |
| RAM consumption | ~100–300 MB | ~100–200 MB |
| Referential integrity | Very strict | Strict (InnoDB) |
| Extensions | Very extensive (PostGIS, pg_trgm, uuid-ossp...) | More limited |
| ChirpStack integration | Natively required | Not supported by ChirpStack |
| Community | Very large, very active | Very large (MySQL) / Large (MariaDB) |
| Owner / commercial | Completely independent and free | MySQL owned by Oracle |

### Justification for the choice: PostgreSQL

- **ChirpStack requirement**: ChirpStack v4 natively requires PostgreSQL as its database. Since FireSense uses ChirpStack, PostgreSQL is a mandatory project dependency. Leveraging this instance to also store the web portal/application user accounts is an efficient decision that avoids deploying an additional engine.
- **Use case in FireSense**: PostgreSQL is used to store IDs, credentials, and data of users registering on the FireSense web portal/application. The relational data model is suitable for this type of structured information (users, roles, sessions, permissions).
- **Advanced JSON support (jsonb)**: PostgreSQL allows storing and indexing JSON documents efficiently with the `jsonb` type, which is useful if the web portal needs to store user configurations or preferences in a flexible format without needing a separate document engine.
- **Independence from Oracle**: MySQL has belonged to Oracle since 2010, generating uncertainty about its future as a free project. PostgreSQL is completely independent, governed by the PostgreSQL Global Development Group, with no owning company.
- **Integrity and reliability**: PostgreSQL is recognized as the most robust open-source relational engine and highly compliant with the SQL standard. For critical data like user accounts, its strict referential integrity and full ACID compliance make it the best option.
- **Useful extensions**: Extensions like `uuid-ossp` (to generate user UUIDs) and `pgcrypto` (to securely manage passwords) are natively available in PostgreSQL without additional configuration.
