# 1. Kubernetes Manifests for Node-RED and Grafana

This guide covers deploying **Node-RED** and **Grafana** on a vanilla Kubernetes cluster with persistent storage, TLS via cert-manager, and Traefik as the ingress controller. It assumes you already have:

- A working `kubeadm` cluster with Calico CNI
- Longhorn installed as default storage class
- cert-manager with a ClusterIssuer (e.g., `letsencrypt-prod`)
- Traefik v3.6 installed and configured as the default IngressClass
- The namespace `iot` created: `kubectl create namespace iot`

All resources are placed in `namespace: iot`. Both applications are served under a single domain (`firesense.example.com`) using sub‑path routing.


---

## 1.1 Node‑RED (StatefulSet)

Node‑RED requires a persistent `/data` directory and a custom `settings.js` file. We inject the configuration via a ConfigMap and use a StatefulSet with a PVC template from Longhorn.

### 1.1.1 Create the Node‑RED ConfigMap

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: nodered-settings
  namespace: iot
data:
  settings.js: |
    module.exports = {
      uiPort: process.env.PORT || 1880,
      httpAdminRoot: '/nodered/',
      httpNodeRoot: '/api/',
      userDir: '/data/',
      flowFile: 'flows.json',
      adminAuth: {
        type: "credentials",
        users: [{
          username: "admin",
          password: "$2a$08$..."   // bcrypt hash, generate with: node-red admin hash-pw
          ,permissions: "*"
        }]
      },
      logging: { console: { level: "info", metrics: false, audit: false } }
    };
EOF
```

Replace the password hash with one generated by `node-red admin hash-pw`.

### 1.1.2 Deploy the StatefulSet and Service

```yaml
# nodered.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: nodered
  namespace: iot
spec:
  serviceName: nodered
  replicas: 1
  selector:
    matchLabels:
      app: nodered
  template:
    metadata:
      labels:
        app: nodered
    spec:
      securityContext:
        fsGroup: 1000
      containers:
        - name: nodered
          image: nodered/node-red:latest
          ports:
            - name: http
              containerPort: 1880
          env:
            - name: TZ
              value: Europe/Madrid
            # Example: passing InfluxDB token and MQTT broker
            - name: INFLUX_TOKEN
              valueFrom:
                secretKeyRef:
                  name: influxdb-secrets
                  key: admin-token
            - name: INFLUX_URL
              value: "http://influxdb:8086"
            - name: INFLUX_ORG
              value: firesense
            - name: MQTT_BROKER
              value: "mqtts://mosquitto:8883"
          volumeMounts:
            - name: data
              mountPath: /data
            - name: settings
              mountPath: /data/settings.js
              subPath: settings.js
          resources:
            requests:
              cpu: 100m
              memory: 256Mi
            limits:
              cpu: 1000m
              memory: 1Gi
          livenessProbe:
            httpGet:
              path: /nodered/
              port: 1880
            initialDelaySeconds: 60
      volumes:
        - name: settings
          configMap:
            name: nodered-settings
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes:
          - ReadWriteOnce
        storageClassName: longhorn
        resources:
          requests:
            storage: 5Gi
---
apiVersion: v1
kind: Service
metadata:
  name: nodered
  namespace: iot
spec:
  selector:
    app: nodered
  ports:
    - port: 1880
      targetPort: 1880
```

```bash
kubectl apply -f nodered.yaml
```

## 1.2 Grafana (Deployment with PVC)

Grafana 11.4 will be served on the `/grafana/` sub-path. We use a `Deployment` with `Recreate` strategy because `ReadWriteOnce` volumes do not support `RollingUpdate`.

### 1.2.1 Secrets and ConfigMaps

Create a secret for the admin credentials:

```bash
kubectl -n iot create secret generic grafana-admin \
  --from-literal=admin-user=admin \
  --from-literal=admin-password='ChangeMe-TFG2026'
```

Create a ConfigMap for the InfluxDB datasource provisioning:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: grafana-datasources
  namespace: iot
data:
  datasources.yaml: |
    apiVersion: 1
    datasources:
      - name: InfluxDB-FireSense
        type: influxdb
        access: proxy
        url: http://influxdb:8086
        jsonData:
          version: Flux
          organization: firesense
          defaultBucket: sensores
        secureJsonData:
          token: ${INFLUX_TOKEN}
```

The token will be injected as an environment variable (see deployment below).

### 1.2.2 PersistentVolumeClaim

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: grafana-data
  namespace: iot
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: longhorn
  resources:
    requests:
      storage: 5Gi
```

### 1.2.3 Deployment and Service

```yaml
# grafana.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: grafana
  namespace: iot
spec:
  replicas: 1
  strategy:
    type: Recreate            # RWO PVC does not allow RollingUpdate
  selector:
    matchLabels:
      app: grafana
  template:
    metadata:
      labels:
        app: grafana
    spec:
      securityContext:
        fsGroup: 472
        runAsUser: 472
      containers:
        - name: grafana
          image: grafana/grafana:11.4.0
          ports:
            - containerPort: 3000
          env:
            - name: GF_SECURITY_ADMIN_USER
              valueFrom:
                secretKeyRef:
                  name: grafana-admin
                  key: admin-user
            - name: GF_SECURITY_ADMIN_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: grafana-admin
                  key: admin-password
            - name: GF_SERVER_DOMAIN
              value: firesense.example.com
            - name: GF_SERVER_ROOT_URL
              value: "https://firesense.example.com/grafana/"
            - name: GF_SERVER_SERVE_FROM_SUB_PATH
              value: "true"
            - name: INFLUX_TOKEN
              valueFrom:
                secretKeyRef:
                  name: influxdb-secrets
                  key: admin-token
          volumeMounts:
            - name: data
              mountPath: /var/lib/grafana
            - name: ds
              mountPath: /etc/grafana/provisioning/datasources
          resources:
            requests:
              cpu: 100m
              memory: 200Mi
            limits:
              cpu: 500m
              memory: 512Mi
      volumes:
        - name: data
          persistentVolumeClaim:
            claimName: grafana-data
        - name: ds
          configMap:
            name: grafana-datasources
---
apiVersion: v1
kind: Service
metadata:
  name: grafana
  namespace: iot
spec:
  selector:
    app: grafana
  ports:
    - port: 3000
      targetPort: 3000
```

```bash
kubectl apply -f grafana.yaml
```

## 1.3 Ingress for HTTP services

Both Node‑RED and Grafana are exposed through the same Traefik ingress, using TLS from Let’s Encrypt.

```yaml
# iot-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: firesense
  namespace: iot
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
    traefik.ingress.kubernetes.io/router.entrypoints: websecure
spec:
  ingressClassName: traefik
  tls:
    - hosts:
        - firesense.example.com
      secretName: firesense-tls
  rules:
    - host: firesense.example.com
      http:
        paths:
          - path: /grafana/
            pathType: Prefix
            backend:
              service:
                name: grafana
                port:
                  number: 3000
          - path: /nodered/
            pathType: Prefix
            backend:
              service:
                name: nodered
                port:
                  number: 1880
          # If you have a frontend service at root:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: nginx-frontend
                port:
                  number: 80
```

**Important:** Change the hostname to your real domain. The TLS secret `firesense-tls` will be automatically created and renewed by cert-manager.

# 2. Kubernetes Manifests for Mosquitto MQTT and InfluxDB

This guide deploys the data backbone of FireSense: **Mosquitto** (MQTT broker) with TLS and **InfluxDB** (time-series database). Prerequisites are the same as the previous guide.

---

## 2.1 Mosquitto MQTT (StatefulSet with TLS)

### 2.1.1 Prepare certificates and password file

You must have your own CA certificate and server certificate/key. Place them in a local `certs/` directory. If you already have them, create the Kubernetes Secrets:

```bash
kubectl -n iot create secret generic mosquitto-tls \
  --from-file=ca.crt=./certs/ca.crt \
  --from-file=server.crt=./certs/server.crt \
  --from-file=server.key=./certs/server.key

# password file for MQTT authentication (plain text, one user:password per line)
kubectl -n iot create secret generic mosquitto-passwd \
  --from-file=passwd=./mosquitto/passwd
```

### 2.1.2 ConfigMap for mosquitto.conf

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: mosquitto-config
  namespace: iot
data:
  mosquitto.conf: |
    persistence true
    persistence_location /mosquitto/data/
    log_dest file /mosquitto/log/mosquitto.log
    log_dest stdout
    log_type all

    listener 1883
    allow_anonymous false
    password_file /mosquitto/passwd/passwd

    listener 8883
    cafile   /mosquitto/certs/ca.crt
    certfile /mosquitto/certs/server.crt
    keyfile  /mosquitto/certs/server.key
    require_certificate false
    tls_version tlsv1.3
```

### 2.1.3 StatefulSet and Service

```yaml
# mosquitto.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mosquitto
  namespace: iot
spec:
  serviceName: mosquitto
  replicas: 1
  selector:
    matchLabels:
      app: mosquitto
  template:
    metadata:
      labels:
        app: mosquitto
    spec:
      securityContext:
        runAsUser: 1883
        fsGroup: 1883
      containers:
        - name: mosquitto
          image: eclipse-mosquitto:2.0.22
          ports:
            - name: mqtt
              containerPort: 1883
            - name: mqtts
              containerPort: 8883
          volumeMounts:
            - name: data
              mountPath: /mosquitto/data
            - name: log
              mountPath: /mosquitto/log
            - name: config
              mountPath: /mosquitto/config
            - name: certs
              mountPath: /mosquitto/certs
              readOnly: true
            - name: passwd
              mountPath: /mosquitto/passwd
              readOnly: true
          resources:
            requests:
              cpu: 50m
              memory: 64Mi
            limits:
              cpu: 500m
              memory: 256Mi
          livenessProbe:
            tcpSocket:
              port: 1883
            initialDelaySeconds: 10
      volumes:
        - name: config
          configMap:
            name: mosquitto-config
        - name: certs
          secret:
            secretName: mosquitto-tls
        - name: passwd
          secret:
            secretName: mosquitto-passwd
            defaultMode: 0640
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes:
          - ReadWriteOnce
        storageClassName: longhorn
        resources:
          requests:
            storage: 2Gi
    - metadata:
        name: log
      spec:
        accessModes:
          - ReadWriteOnce
        storageClassName: longhorn-single-replica
        resources:
          requests:
            storage: 1Gi
---
apiVersion: v1
kind: Service
metadata:
  name: mosquitto
  namespace: iot
spec:
  type: LoadBalancer                  # MetalLB will assign an IP from the pool
  loadBalancerIP: 10.0.0.241          # Choose a fixed IP from your MetalLB range
  selector:
    app: mosquitto
  ports:
    - name: mqtt
      port: 1883
      targetPort: 1883
    - name: mqtts
      port: 8883
      targetPort: 8883
```

```bash
kubectl apply -f mosquitto.yaml
```

**Why LoadBalancer?** MQTT is not HTTP; it cannot be routed by an Ingress. A `LoadBalancer` service with a fixed IP from MetalLB exposes the broker directly to your IoT devices.

# 3. Deploying Harbor Registry (v2.15.0)

This guide deploys **Harbor**, the CNCF-graduated container registry, on your Kubernetes cluster. It includes Trivy scanning, Cosign signing, internal TLS, and persistent storage on Longhorn. It assumes:

- cert-manager with a `letsencrypt-prod` ClusterIssuer
- Traefik as the IngressClass
- Longhorn as storage backend
- A dedicated namespace: `kubectl create namespace harbor`

---

## 3.1 Create admin secret

```bash
kubectl -n harbor create secret generic harbor-admin-secret \
  --from-literal=HARBOR_ADMIN_PASSWORD='SuperSecreto2026!'
```

## 3.2 Prepare Helm values file

Save the following as `values-harbor.yaml`. It configures persistence on Longhorn, ingress with TLS, internal TLS, and resource limits suitable for a 24 GB cluster.

```yaml
# values-harbor.yaml — Harbor 2.15.0 on Longhorn + Traefik + cert-manager
expose:
  type: ingress
  tls:
    enabled: true
    certSource: secret
    secret:
      secretName: harbor-tls
  ingress:
    hosts:
      core: harbor.firesense.example.com
    className: traefik
    annotations:
      cert-manager.io/cluster-issuer: letsencrypt-prod
      traefik.ingress.kubernetes.io/router.entrypoints: websecure

externalURL: https://harbor.firesense.example.com
existingSecretAdminPassword: harbor-admin-secret
existingSecretAdminPasswordKey: HARBOR_ADMIN_PASSWORD

internalTLS:
  enabled: true
  certSource: auto

persistence:
  enabled: true
  resourcePolicy: keep
  persistentVolumeClaim:
    registry:
      storageClass: longhorn-replicated
      accessMode: ReadWriteOnce
      size: 100Gi
    jobservice:
      jobLog:
        storageClass: longhorn
        accessMode: ReadWriteOnce
        size: 1Gi
    database:
      storageClass: longhorn-replicated
      accessMode: ReadWriteOnce
      size: 5Gi
    redis:
      storageClass: longhorn
      accessMode: ReadWriteOnce
      size: 1Gi
    trivy:
      storageClass: longhorn
      accessMode: ReadWriteOnce
      size: 10Gi

updateStrategy:
  type: Recreate    # required with ReadWriteOnce volumes

core:
  replicas: 1
  resources:
    requests:
      cpu: 100m
      memory: 256Mi
    limits:
      cpu: 1
      memory: 1Gi
jobservice:
  replicas: 1
  resources:
    requests:
      cpu: 100m
      memory: 256Mi
    limits:
      cpu: 500m
      memory: 512Mi
registry:
  replicas: 1
  registry:

      requests:
        cpu: 100m

      limits:
        cpu: 1

  controller:
    resources:
```

## 3.4 Post‑installation setup

1. **Access the UI** at `https://harbor.firesense.example.com`. Log in with:
   - Username: `admin`
   - Password: `SuperSecreto2026!`
2. **Change the admin password** immediately (User Profile → Change Password).
3. **Create a private project** named `firesense`:
   - Enable “Auto scan on push”
   - Enable “Prevent vulnerable images from running”
   - Set severity threshold to `High`
4. **Create a robot account** for Kubernetes image pulls:
   - Name: `k8s-puller`
   - Expiration: `-1` (never)
   - Permissions: pull only on the `firesense` project
   - **Copy the token** – it will not be shown again.
5. **Configure tag retention**: In the project, add a rule “retain last 10 most-recently-pushed images” (weekly).
6. **Tag immutability**: Optionally, add a rule to make tags matching `v*` and `release-*` immutable.

## 3.5 Connect your Kubernetes cluster to Harbor

Create a `docker-registry` secret in every namespace that will use images from Harbor:

```bash
kubectl -n iot create secret docker-registry harbor-creds \
  --docker-server=harbor.firesense.example.com \
  --docker-username='robot$firesense+k8s-puller' \
  --docker-password='<ROBOT_TOKEN>' \
  --docker-email=admin@firesense.example.com
```

Then patch the default ServiceAccount in the namespace so all pods inherit the pull secret automatically:

```bash
kubectl -n iot patch serviceaccount default \
  -p '{"imagePullSecrets":[{"name":"harbor-creds"}]}'
```

Now you can push images to Harbor, sign them with Cosign, and deploy them to your cluster.

---

## 3.6 Image signing (Cosign)

For supply-chain security, sign your images before pushing:

```bash
# Generate a key pair (first time)
cosign generate-key-pair

# Sign an image
cosign sign --key cosign.key harbor.firesense.example.com/firesense/firesense-bot:v1.0
```

Harbor will display the signature verification status in the UI.

---

## 3.7 Resource footprint

Harbor with all services uses roughly:
- **Idle**: ~2 GB RAM
- **During a Trivy scan**: up to 5 GB RAM

Schedule scans for off‑peak hours to avoid contention with your IoT workloads. With the given values, a single replica of each component is enough for a development/TFG environment.
